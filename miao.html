Download & install

Download from the github page. Click â€˜view rawâ€™ and save the file to a location of your choice.

The install procedure varies according to your system. Most modern operating systems will have an â€œinstallâ€ button on the window that appears when you open the font.

In Windows XP you need to copy the file into the C: > Windows > Fonts directory.

After installing, check that itâ€™s working by restarting your browser and going to Alan Woodâ€™s test page. This is also a helpful reference resource. If all of the characters display, the font has probably installed correctly.

As Alan Wood suggests, it can be helpful to compare the characters which appear on the test page with the Official Unicode chart (pdf).

The font can only be used properly in an application which supports Graphite. A list of these is given on the Graphite website. (Note that Microsoft Word cannot display Miao Unicode correctly.)

For most users the most important application which supports Graphite will probably be the word processor LibreOffice Writer.

For Linux users, it is possible to expand the number of Graphite-supporting applications by installing the pango-graphite package. (For example, this will enable Graphite support in gedit and Inkscape.)

Graphite features

The font relies on Graphite programming to work, but Graphite also gives scope for aesthetic adjustment and stylistic features.

For those interested in using the Latin-script features shown above, you can enable them in LibreOffice/OpenOffice by using the following codes in the font field:

  • Miao Unicode:smcp=1 for small caps
  • Miao Unicode:c2sc=1 for all small caps, including numerals and some punctuation
  • Miao Unicode:onum=1 for old-style figures

Known issues

While Miao Unicode is fully functional for use, it is still in a phase of testing and development.

(You do not need to worry about future versions breaking compatibility, however. Documents created using this font will always display correctly with future versions of the font as well, as indeed they should display correctly with any font based on the Unicode implementation instructions.)

These are some of the issues which I am aware of:

  • The font lacks the stylistic variant for the â€˜wartâ€™ (a translated technical term!) in characters like ğ–¼, ğ–¼…, and ğ–¼¾. There should be an option to render this as a dot on the right of the glyph for the Chuxiong orthography, as shown in point 5 here (pdf).
  • Various workarounds have been employed to make the finals as legible as possible, including using smaller variants of the finals for above and below position. This has the side effect of giving a slightly inconsistent appearance.
  • Currently a section of the PUA is used for the compressed forms. It would probably be better to re-encode these glyphs without Unicode values and access them in Graphite via post-script names.
  • The spacing is extremely approximate. The side bearings and kerning values all need to be fine-tuned.
  • The outlines of many of the finals are less-than-optimally drawn.
  • Some of the Graphite code seems â€œprecariousâ€: it works, but doesnâ€™t seem like a satisfying solution. This will probably be re-written gradually as the font is developed.
  • It is built on a compiled version of Adobe Source Sans Pro, rather than on its source. This does not appear to have any adverse effects (all information, including hinting instructions, is retained), but it is clearly not ideal.
  • For some reason the OpenType style set names are interfering with the copyright field in the font, so that the copyright often displays as â€œstraight lâ€ or â€œslashed zeroâ€ instead of the intended message about the OFL license etc. I have not been able to ascertain why this is happening.
